# Why You Get 403 on Auto-Swipe

The error happens on **get_candidates** (discovery feed), which is the first request in the auto-swipe flow.

## Root cause: wrong GraphQL path

- **What we send:** `POST https://e2p-okapi.api.okcupid.com/graphql/`
- **What the browser uses:** Operation-specific paths, e.g. `POST .../graphql/WebUserVote` for vote.

OkCupid’s API does **not** accept a generic `/graphql/` endpoint. It expects a path per operation, e.g.:

- Vote (like/pass): `/graphql/WebUserVote`
- Discovery (profiles to swipe): **unknown until you capture it** – e.g. `/graphql/DiscoveryFeed` or similar.

In `sample.json`, **`graphql.discovery_path`** is empty, so the client falls back to `/graphql/`. The server responds with **403 Forbidden** for that path.

## What to do

1. **Capture the discovery path**
   - Open OkCupid, go to the swipe stack (DoubleTake / discovery).
   - In DevTools → Network, find the request that returns the list of profiles (cards).
   - Check **Request URL** or **:path** (e.g. `/graphql/DiscoveryFeed` or `/graphql/DoubleTake`).

2. **Set it in sample.json**
   - Set `graphql.discovery_path` to that path (e.g. `"/graphql/DiscoveryFeed"`).

3. **Use the real query**
   - In the same request, copy the **Payload** → `query` (and `operationName`, `variables`).
   - Put the exact query into `okcupid_api/graphql_operations.py` as `DISCOVERY_FEED_QUERY` (and adjust variable names if needed).

After setting the correct **discovery_path** and **query**, the discovery request should no longer return 403 (assuming auth and cookies are valid).

## Other possible causes of 403

- **Expired or invalid token/cookies** – refresh from DevTools and update `sample.json`.
- **Missing or wrong headers** – e.g. `Origin: https://www.okcupid.com`, `Referer`, and all `X-Okcupid-*` headers must match the browser (already set from your screenshot).
- **Cloudflare / bot detection** – the server may block non-browser clients; matching path + query + headers gives the best chance.
